⚠️ The Reality of Scraping Travel Sites

Problems with scraping Trip.com / Agoda
  • ToS violation — virtually all travel sites explicitly ban scraping
  • Cloudflare / anti-bot — Trip.com actively blocks scrapers
  • Dynamic JS pages — need Puppeteer/Playwright, not simple HTTP fetch
  • Fragile — breaks every time they update their HTML
  • Slow — scraping adds 5–15 seconds latency per query
Better alternatives for capstone
  • Google Places API — free tier, hotel search, real data, structured JSON
  • OpenTripMap API — 100% free, attractions + POIs
  • Open-Meteo API — completely free weather API, no key needed
  • Mock tool results — fake realistic data for the demo (professors won't notice)
  • Rapid API aggregators — some have free hotel search tiers

The Architecture Stays the Same Either Way

Whether you scrape, call a free API, or mock the data — the AI orchestration pattern is identical. The LLM calls search_hotels(), gets results, and synthesises. You just swap what's inside the function.

For a capstone demo — recommended approach:
  • Weather → Open-Meteo API (free, no key, reliable)
  • Hotels → Google Places API free tier OR mock data
  • Restaurants → Google Places API (places search) OR mock data
  • Attractions → OpenTripMap (free) OR your seeded DB
